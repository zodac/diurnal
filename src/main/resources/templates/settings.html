Because Account and Preferences are grouped in the left-column wrapper (needed for the desktop column), that wrapper is `display:contents` below `lg` (`contents lg:flex…`): its box disappears on mobile so Account and Preferences become direct flex items of THIS container alongside Appearance, and `order-last` on Preferences drops it to the end — giving Account, Appearance, Preferences. At `lg` the wrapper is a real flex column again and the order resets, so the desktop layout is unchanged. Each preference control saves ITSELF via its own PATCH endpoint (see the hx-patch on the radiogroups below and on the two selects in partials/select-field): updating one setting no longer touches the others, so there is no wrapper form and no shared/scoped trigger. This wrapper is just the layout container (and the delegation point for the "Saved" indicator in the script below). !}

{! Password (local accounts only). Mirrors the Display Name field's view/edit chrome (hover-revealed Edit, fixed-height ringed edit state) so switching modes never resizes the row. Changing the password is a two-step flow that keeps every control in the SAME place: 1. hover → Edit appears; 2. Edit → the value becomes an editable input with Save/Cancel; 3. Save validates only that it is non-empty; 4. on success the input is replaced IN PLACE by a "confirm" input asking for the password again — only a matching re-entry actually persists the change. Each input hides its value and carries an eye/reveal toggle. NOTE: passwords are stored only as Argon2id hashes and cannot be recovered, so the resting view shows a fixed placeholder — there is no real password to reveal there, so the read row has no eye toggle (the reveal on the edit and confirm inputs below works normally, as those hold a real value). SECURITY: the change now starts by asking for the CURRENT password, so a hijacked session cannot silently reset it. Each of the three steps (current → new → re-enter) occupies the SAME slot, so the row never grows or reflows. !} {#if canChangePassword}
