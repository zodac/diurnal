{! A fixed-size, top-cropped screenshot thumbnail — the reusable UI element for ANY settings preview thumbnail. It guarantees a consistent size per viewport (see `.preview-thumb` in app.css for the ratios) by cropping whatever screenshot it is given to the TOP of a fixed frame, so the real aspect ratio of the source image is irrelevant. Drop this in wherever a settings thumbnail is needed; to resize every thumbnail, change `.preview-thumb` once. Two modes: • Calendar/plain mode (default): renders the four responsive `` variants for ONE screenshot — web/mobile × light/dark (when imgDark is given). • Theme mode (`themeBase` set to "light" | "dark" | "system"): the theme picker needs one full-page shot PER calendar style. This renders all three (`theme-{style}-{themeBase}{,-mobile}.png`) and lets the page show only the one matching the selected style — each style's pair is wrapped in a `[data-cal-style]` group, toggled by the `#theme-options[data-cal=…]` CSS in app.css (settings.html keeps `data-cal` in sync with the Calendar-style radios). Theme shots are theme-specific (no dark variant). LOADING / SWITCHING. Two separate hiding mechanisms, on purpose: • VIEWPORT is gated with `display` (the `sm:` classes): the OTHER viewport's images are `display:none` and `loading="lazy"`, so they are never fetched — we don't download what the user can't see (a desktop user never sees the mobile thumbnails, or vice-versa, unless the window crosses `sm`). • The not-yet-shown variant WITHIN the current viewport — the opposite theme here, or the other calendar styles in theme mode — is hidden with `visibility`/`invisible` (NOT display), so it stays laid out and settings.html can `decode()` it up front. That makes theme / calendar-style switches flash-free; a `display:none` image cannot be pre-decoded (its decode() promise never resolves).
