{! Shared pagination footer for every data table. Params: page — paginated result with currentPage / totalPages / totalCount / items pageUrl — full-page URL for the non-JS href fallback (e.g. "/actions") listUrl — HTMX partial URL swapped into `target` (e.g. "/actions/list") target — CSS selector that receives the swapped list (e.g. "#action-list") extraQuery — optional pre-encoded query suffix, e.g. "&q=foo" (pass "" when unused) alwaysShow — true to render the footer even on a single page (real .dt-tables do this); false to only render it when there's more than one page (read-only lists) borderless — true to drop the top border/divider (for a grid of standalone cards, not a clipped `.dt-table`, where a full-width rule above the footer reads as stray); false/omitted keeps the default table divider The "Showing" counter exposes #showing-shown / #showing-total so surgical (HTMX 204) row removals can keep the count in sync without re-rendering the whole list. `alwaysShow=true` (Actions, Admin — real `.dt-table`s in a rounded `overflow-hidden` card) keeps the footer rendered even on a single page, for two reasons: it stops the last table row sitting on the card's rounded bottom corner (where the edit/confirm highlight ring would be clipped), and it guarantees the #showing-shown / #showing-total spans always exist, so the surgical (HTMX) add/delete handlers can keep the count in sync across the empty↔non-empty transition (e.g. adding the very first row). Such a list shows "Showing 0 of 0" when empty. `alwaysShow=false` (Stats — a grid of standalone cards, not a clipped table, with no editable rows) keeps the original behaviour: no footer at all on a single page, just the page-navigation controls once the list spills onto more than one page. Either way the page-navigation block (Previous / Page X of Y / Next) only appears when there's more than one page. !} {#if alwaysShow || page.totalPages > 1}
